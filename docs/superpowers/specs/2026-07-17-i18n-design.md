# Spec: i18n (DE/EN)

**Datum:** 2026-07-17 · **Status:** genehmigt (Brainstorming mit Jay, autonome Umsetzung freigegeben)

## 1. Ziel & Kontext

Jay fährt deutsches Obsidian; das Plugin zeigt bisher nur englische UI-Chrome-Texte
(`src/core/strings.ts`, `STRINGS`-Objekt, Kommentar dort: „i18n-Ausbau später"). Dieses Spec setzt
den Ausbau um: automatische Sprachumschaltung EN/DE nach Obsidians UI-Sprache, ohne manuelles
Override-Setting (Entscheidung im Brainstorming: YAGNI, kein Präzedenzfall im Kit-Muster für
UI-Chrome-Override — yijings `readingLang` ist ein anderer Fall, siehe §6).

**Kit-first-Befund:** `obsidian-kit/src/pure/i18n.ts` löst genau das (EN/DE, Fallback,
`{0}`-Interpolation) und ist bereits dreifach vendored (vault-crews, obsidian-paperize,
yijing-oracle) — Regel-der-Drei längst erreicht. Hier: **vierte Vendoring-Instanz**, kein neuer
Code in der Engine.

**Explizit außerhalb dieses Scopes:** Robustheits-Block (Backlog 4/5/6, eigene Session),
Kit-Extraktion der i18n-Engine (läuft separat über `/drift-audit`).

## 2. Architektur

- **`src/vendor/kit/i18n.ts`** (neu) — 1:1-Kopie aus `obsidian-kit/src/pure/i18n.ts`. Obsidian-frei,
  vom Pure-Core-Gate (`scripts/check-pure.mjs`) erfasst. Exporte: `Lang`, `pickLang`, `setLang`,
  `getLang`, `defineStrings`, `t`.
- **`src/i18n/strings.ts`** (neu, ersetzt `src/core/strings.ts`) — exportiert `EN`, `DE` (je
  `Record<string, string>`) und `registerI18n()` (ruft `defineStrings({ en: EN, de: DE })`).
- **`src/main.ts`** `onload()`: `registerI18n(); setLang(pickLang(getLanguage()));` als **erste**
  Zeilen nach dem Settings-Laden (`sanitizeSettings(mergeSettings(...))`), **vor**
  `addSettingTab`/`registerView`/`addRibbonIcon`/`addCommand` — sonst rendern die ersten
  `t()`-Aufrufe rohe Keys (gleiche Reihenfolge-Regel wie yijing-oracle, PROF-OBS-07).
  `getLanguage` kommt aus `obsidian` (ab 1.8.0; `manifest.json` `minAppVersion` ist hier bereits
  1.8.7 ✓, kein Anhebungsbedarf).
- Alle Aufrufer importieren `t` aus `./vendor/kit/i18n` (Tiefe je nach Datei). Betroffen:
  `src/main.ts`, `src/core/viewmodel.ts`, `src/obsidian/{confirm-modal,generate-panel,
  history-panel,preset-editor,settings-tab,view}.ts`.
- **Pure-Core-Schnitt bleibt gewahrt:** `core/viewmodel.ts` darf `t()` weiterhin aufrufen, weil
  `vendor/kit/i18n.ts` selbst obsidian-frei ist — exaktes Gegenstück zu yijings
  `core/settings.ts`, das `pickLang` aus derselben vendorten Datei importiert.
- **Keine `data.json`-Änderung, keine Migration.** Reine Code-Umstellung; `LigSettings` bekommt
  kein neues Feld (§ Sprachwahl-Entscheidung oben).

## 3. Sonderfälle

**Parametrisierte Strings** (bisher Funktionen in `STRINGS`, z.B.
`statusDownloading: (pct) => \`Downloading model… ${pct}%\``) werden zu Keys mit `{0}`/`{1}`-
Platzhaltern; Call-Sites rufen `t("status.downloading", pct)` statt `STRINGS.statusDownloading(pct)`.
Bei `noteFailed` (Original-Signatur `(msg, imagePath)`, Text nennt aber `imagePath` zuerst) wird die
Argumentreihenfolge am Call-Site an die Lesereihenfolge angepasst: `t("notice.noteFailed",
imagePath, msg)` mit Text `"Image saved to {0}, but the note failed: {1}"`.

**Pluralisierung** (`historyVariations`, bisher `n === 1 ? "1 variation" : \`${n} variations\`"`):
Die Kit-Engine kennt keine ICU-Pluralregeln (nur `{n}`-Positional-Interpolation) — dafür gibt es im
Repo keinen Präzedenzfall, einen einzelnen Call-Site rechtfertigt keine neue Abstraktion. Lösung:
**zwei Keys** (`history.variations.one`, `history.variations.other`), die bestehende Ternary am
Call-Site bleibt (`n === 1 ? t("history.variations.one") : t("history.variations.other", n)`) —
minimaler Diff, kein neuer Mechanismus.

**Zwei hartkodierte Strings außerhalb von `STRINGS`** (Sweep-Fund, `settings-tab.ts:78,88`):
`"Downloaded"` (Tooltip auf dem Erledigt-Icon) und `"Model downloaded"` (Notice nach erfolgreichem
Download) waren nie Teil von `STRINGS` und würden ohne diesen Fund englisch hartkodiert bleiben.
Bekommen eigene Keys (`settings.model.downloadedTooltip`, `notice.modelDownloaded`).

**Absichtlich unübersetzt:** `"Seed"` (Feld-Label) bleibt in beiden Sprachen `"Seed"` — im
Stable-Diffusion-Kontext auch im deutschsprachigen Raum durchgehend der gebräuchliche Begriff,
keine Lücke. `"SD-Turbo (ONNX, fp16)"` (Settings-Überschrift) ist ein Produktname, kein UI-Text —
bleibt unverändert, kein `STRINGS`-Eintrag.

**Dezimaltrennzeichen (bewusste Nicht-Lokalisierung):** `settings.model.download` interpoliert den
per `.toFixed(1)` berechneten GB-Wert (`{0}`) unverändert mit Punkt (z.B. „2.5"), auch auf Deutsch —
`Intl.NumberFormat`-Lokalisierung für diesen einen Wert wäre Scope-Creep gegenüber einem reinen
String-Sweep. Die statischen `~2.5 GB`-Erwähnungen in `empty.noModelCta` und
`settings.model.deleteConfirm` sind dagegen literale Textbausteine (kein interpolierter Wert) und
bekommen wie der Rest der DE-Übersetzung ein Komma (`~2,5 GB`) — beide Quellen standen schon vor
diesem Spec unabhängig nebeneinander (kein neues Problem, nur nicht rückwirkend behoben).

## 5. Sweep-Scope (Dateien)

`src/main.ts` (Wiring + `onload`-Reihenfolge + `STRINGS`→`t()`), `src/core/viewmodel.ts`
(Statuszeilen-Texte), `src/obsidian/confirm-modal.ts`, `src/obsidian/generate-panel.ts`,
`src/obsidian/history-panel.ts`, `src/obsidian/preset-editor.ts`, `src/obsidian/settings-tab.ts`
(inkl. der zwei hartkodierten Strings), `src/obsidian/view.ts`. `src/core/strings.ts` wird gelöscht
(ersetzt durch `src/i18n/strings.ts`).

**Nicht betroffen:** `src/core/pipeline/*` (keine UI-Strings), Modell-Dateinamen/technische Logs.

## 6. Tests

Neue Datei `tests/obsidian/i18n.test.ts` (Muster: vault-crews' gleichnamige Datei):

- `registerI18n()` + `t()`: bekannter EN-Key liefert korrekten Text; `setLang("de")` schaltet um;
  unbekannter Key fällt auf den Key selbst zurück; `{0}`/`{1}`-Interpolation inkl. unbesetzter
  Platzhalter.
- **EN/DE-Key-Parität:** `Object.keys(DE).sort()` === `Object.keys(EN).sort()` — verhindert Drift
  zwischen den Sprachen (kein Key nur in einer Sprache definiert).
- Stichprobe: zentrale Keys (`cmd.open`, `view.title`, `generate.button.generate`) sind in beiden
  Dicts nicht-leer.

Die Kit-Engine selbst (`vendor/kit/i18n.ts`) wird **nicht** erneut getestet — das leistet bereits
`obsidian-kit/tests/i18n.test.ts`, hier nur vendored. Bestehende Tests, die `STRINGS` direkt
importieren (falls vorhanden), werden auf `t("…")`-Erwartungen umgestellt.

## 7. Abgrenzung zu yijings `readingLang`

Zur Klarheit, warum hier **kein** Sprach-Setting: yijing-oracles `readingLang` (auto/en/de)
steuert die Sprache des **Orakel-Lesetexts** (Inhalt, nicht Chrome) — dort gibt es einen echten
Grund für manuelle Wahl (z.B. englische Reading-Texte in einem sonst deutschen Vault). Hier gibt es
keinen Inhalt dieser Art, nur UI-Chrome; die UI-Chrome-Sprache folgt in allen drei Referenz-Plugins
ausnahmslos automatisch `getLanguage()`, ohne Override.
